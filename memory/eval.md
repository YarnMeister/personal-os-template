# Pre-Response Quality Checklist · Updated: 2026-06-21

Run this checklist before every substantive response. A substantive response is any recommendation, strategy, prioritisation, or decision that the user may act on.

## Checklist

- [ ] **Read context/active.md** — Have I loaded the current sprint and top-3 priorities? Does this response align with or acknowledge them?
- [ ] **Check memory/learnings.md** — Are there any relevant patterns or corrections from previous sessions that apply to this request?
- [ ] **Ground in context/org.md** — Am I using the correct org terminology? (Check the glossary for ambiguous terms like "the community" or "the team".) Are my stakeholder references accurate?
- [ ] **Role-specific recommendations** — Are my recommendations specific to the Head of Product Operations role? Scoped to PM Ops / Design Ops / TPM Ops context? Not generic advice.
- [ ] **Named owners and dates** — Do all next steps have a named owner and a target date? "Someone should do this sometime" fails this check.
- [ ] **Staleness check** — Is context/active.md more than 2 working days old? If so, surface a warning before responding. Are there any memory/learnings.md entries older than 30 days? Flag them for review.
- [ ] **Sensitivity check** — Does any content in my response belong in a protected enterprise system? (Personnel data, unreleased roadmap, commercial terms.) Flag before writing to any file.

## When to skip

Skip for: casual questions, clarifications, one-liner factual lookups, and conversational exchanges that will not be acted on. When in doubt, run it.

## Staleness thresholds

- `context/active.md` — warn if `Updated:` date is more than 2 working days ago
- `memory/learnings.md` entries — flag any entry whose date prefix is more than 30 days before today
- `memory/tools.md` — flag if `Updated:` date is more than 30 days ago (M3+)
