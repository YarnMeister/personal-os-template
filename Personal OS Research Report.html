Executive Summary

The core problem — scattered markdown, repeated context lookups, no structured memory — is real and solvable. The good news: you don't need to invent anything. A small cluster of open-source projects and emerging community patterns has already converged on a workable architecture. The bad news: most documented examples are built for developers using the Claude Code CLI, which is explicitly out of scope here. The adaptation work is genuine but tractable.

The recommendation is a Layered Context Engineering approach — a three-tier markdown file hierarchy (Constitution → Living Memory → Project Brains) combined with SKILL.md-based workflows, stored in a GitHub-backed repository, surfaced through VS Code's native AI chat. This pattern works with any AI assistant embedded in VS Code (Copilot, Codeium, Continue.dev, etc.), requires zero custom code, and is already partially documented in the community.

The central design tension — capability vs. simplicity for non-technical users — resolves in favour of simplicity at every decision point. A system that gets used by 8 out of 10 team members at 60% depth beats a sophisticated system used by 2 out of 10 at full depth. Build the floor before the ceiling.

Recommended: Layered Context Engineering — 3-tier markdown + SKILL.md + GitHub versioning

A
Single Flat Instruction File
One file: CLAUDE.md or copilot-instructions.md

One instruction file at the workspace root. Everything the AI needs to know — persona, context, rules, project background — in a single markdown document. Section 03

Recommended Approach
In-Depth

Layered Context Engineering — a three-tier markdown hierarchy + SKILL.md workflows + GitHub versioning. Here's how to build it, what goes where, and how to maintain it.

Architecture Overview

The Three-Tier Model

Each tier answers a different question. Tier 1 is always loaded. Tiers 2 and 3 are loaded on demand or via explicit reference. Skills are loaded by the AI when a matching task is detected.

personal-os/ ├── CLAUDE.md ← Tier 1: Constitution (auto-loaded by VS Code) ├── AGENTS.md ← Alias / same content (Copilot reads AGENTS.md) ├── memory/ │ ├── learnings.md ← Tier 2: Living Memory (AI appends each session) │ ├── decisions.md ← Key decisions log (auto-updated) │ └── tools.md ← MCP/tool capability registry ├── context/ │ ├── me.md ← Who I am, how I work, org context │ ├── org.md ← Team structure, key stakeholders, priorities │ └── active.md ← Tier 2: Active sprint / current focus ├── projects/ ← Tier 3: One file per active project │ ├── q3-roadmap-review.md │ └── team-tooling-audit.md ├── areas/ ← Tier 3: Ongoing responsibilities │ ├── product-community.md │ └── okr-tracking.md ├── knowledge/ ← Reference: meeting notes, decisions, research ├── .github/skills/ ← SKILL.md files (project-level, per agentskills.io spec) │ ├── writing-prds/ │ │ └── SKILL.md │ ├── stakeholder-alignment/ │ │ └── SKILL.md │ └── product-operations/ ← From Lenny Skills repo (MIT licensed) │ └── SKILL.md ├── templates/ ← Meeting notes, project brief, decision log ├── BACKLOG.md ← Brain-dump inbox (from PersonalOS) └── .vscode/mcp.json ← Shared MCP config (committed to repo)

What Goes Where

1️⃣
Tier 1 — Constitution
CLAUDE.md · Always loaded · ~200 lines max

Your standing orders. Never put content here that can go somewhere else.

  • Who you are and what this workspace is for
  • Hard rules (tone, confidentiality, what to never do)
  • A routing table: "For active project context read projects/<name>.md"
  • Reference to memory/tools.md for available tools
  • Reference to context/active.md for current sprint focus

Key discipline: If this file exceeds 200 lines, something is in the wrong tier. The file is a hub, not a dump.

2️⃣
Tier 2 — Living Memory
memory/ + context/active.md · AI updates after each session

What accumulates over time. Instruct the AI (in Tier 1) to append to memory/learnings.md at the end of every meaningful conversation — one line per new pattern or correction discovered.

  • learnings.md — patterns the AI has learned about how you work
  • decisions.md — key decisions with context (date, alternatives rejected)
  • tools.md — every MCP source, what it's good for, rate limit notes
  • context/active.md — current sprint, top 3 priorities, open questions

Key discipline: learnings.md and active.md are the two files you review weekly. The others are append-only logs.

3️⃣
Tier 3 — Project Brains
projects/ + areas/ · Loaded per project

Deep context that only loads when you're working on a specific project. One file per project or area. Include everything the AI needs to not ask twice: background, constraints, decisions made, key stakeholders, current status.

  • Project brief (what, why, who)
  • Current status and next actions
  • Key decisions and their rationale
  • Relevant stakeholders and their positions
  • Links/references to Confluence, Glean, or Miro artefacts

Trigger phrase: "Read projects/q3-roadmap-review.md then help me with..." — reduces Glean/Confluence round-trips to one per project setup.

The Skills Layer

Skills are repeatable workflows stored as SKILL.md files. The AI loads the relevant skill automatically when it detects a matching task — or you invoke it directly. As of April 2026, VS Code's GitHub Copilot agent mode reads SKILL.md files natively.

Starter skill set for Product Operations: Fork the RefoundAI/lenny-skills repo and copy the following skills into your skills/ directory: product-operations, writing-prds, stakeholder-alignment, prioritizing-roadmap, setting-okrs-goals, running-effective-meetings, cross-functional-collaboration. These 7 skills cover the core Product Ops workflow and require no modification to use immediately.

Reducing MCP Round-Trips

The cost of live Glean/Confluence/Miro lookups compounds fast. The recommended pattern: consolidate once, reference often.

1

Periodic consolidation sessions (weekly)

Once a week: "Read the last 5 Confluence pages I've worked on and summarise the key decisions and open questions into knowledge/this-week.md." This turns expensive live lookups into cheap local reads for the rest of the week.

2

Project Brain as Glean/Confluence proxy

When starting a new project, do one thorough Glean/Confluence sweep and write the findings into a Project Brain file (projects/<name>.md). That file becomes the local reference for the project lifetime — MCP tools are only needed for real-time changes.

3

tools.md as query router

Document in memory/tools.md which MCP source is authoritative for what: "Glean = enterprise search for documents; Confluence = team wiki for decisions and specs; Miro = visual artefacts and workshop outputs." The AI routes queries to the right source instead of querying all sources redundantly.

4

Glean MCP — context-aware query routing

Glean's Enterprise Graph maps relationships across your apps, documents, people, and conversations, and its MCP server was benchmarked at 2.5× preferred over off-the-shelf MCP tools using 30% fewer tokens. Configure the Glean MCP for VS Code using the built-in MCP Configurator under Settings → Install tab — it generates the config snippet and handles OAuth. Document in memory/tools.md: "Glean = enterprise search for documents and people; route discovery queries here first before Confluence." This query routing discipline is what prevents unnecessary Confluence and Miro round-trips.

The Self-Improving Loop

Borrowed from Peter Yang's AI advisor system. Include an eval.md file that the AI checks before responding to substantive requests:

eval.md — Quality checklist (include in SKILL.md or root)
## Pre-Response Checklist
Before responding, verify:
- [ ] Did you read context/active.md for current priorities?
- [ ] Did you check memory/learnings.md for known patterns?
- [ ] Is your response grounded in context/org.md stakeholder map?
- [ ] Are recommendations specific to this role, not generic?
- [ ] Are next steps actionable with named owners and dates?
Only respond when all checks pass.

Section 05

UX & Design Considerations

What makes these systems actually get used vs. abandoned — with specific attention to non-technical adopters in an enterprise setting.

🚦
The Adoption Paradox

Knowledge management systems fail not because they lack content, but because people don't use what's already there. The three failure modes, in order of frequency: (1) unclear whether content is current, (2) too slow to find anything, (3) easier to just ask someone. A Personal OS has a unique advantage here — the AI does the finding, so (2) and (3) are neutralised. The critical remaining risk is (1): staleness.

Design principle: Every file needs a clear "owner + last updated" convention — even one line at the top: # Active Context · Updated: [date]. The AI can enforce this as a check in the eval.md checklist.

🎯
One Inbox to Rule Them All

The PersonalOS BACKLOG.md brain-dump pattern is the highest-value UX decision in the system. Non-technical users need a zero-friction capture point — somewhere to dump thoughts without caring about structure. The AI processes and routes the dump; the user never files anything.

Key instruction in CLAUDE.md: "When asked to 'process my backlog', read BACKLOG.md and route items to the appropriate project file, knowledge file, or action item. Then clear BACKLOG.md." This single workflow replaces most of the manual filing overhead.

⚡
The 5-Minute Daily Ritual

The single highest-leverage habit is a 5-minute morning session: "Read context/active.md and tell me my top 3 priorities for today." This session also updates active.md for the day. Systems that require more than 5 minutes of daily maintenance are abandoned within 3 weeks by non-technical users (consistent finding across KM adoption research). Design the daily ritual to fit within that window.

Morning prompt template: Include this as a SKILL.md — "morning-standup" — so it's invocable with a single phrase.

🔗
Language Consistency = Context Quality

When users describe the same concept using different terms across files, the AI has to resolve ambiguity — which it does poorly. Establish a shared vocabulary in context/org.md: "When I say 'the community', I mean the Product Community of ~12 people." This eliminates a major source of poor AI responses in knowledge work contexts.

Practical pattern: Add a "Glossary" section to context/org.md. Have the AI flag whenever it encounters ambiguous terminology and suggest a canonical term to add to the glossary.

📊
Information Architecture for Non-Technical Users

Non-technical users navigate by meaning, not by file path. Name files descriptively, not technically: q3-roadmap-review.md not project-003.md. Use emoji prefixes for visual scanning in the VS Code file tree: 📋 active.md, 🧠 learnings.md. Create a README.md at the root that explains the system in plain language — this also doubles as the AI's "map" of the workspace.

⚠️
Common Pitfalls
  • Over-structuring at setup: Design the folder structure after a week of real use, not before. Let the AI suggest structure from the BACKLOG.
  • Instruction file bloat: CLAUDE.md over 200 lines is the most common failure mode — it signals that the tiered model isn't being respected.
  • Treating memory as a database: Memory files are curation surfaces, not logs. Append and prune; never let them grow unbounded.
  • Skipping the eval checklist: Without eval.md quality gates, AI responses drift to generic over time as context accumulates.

MCP Configuration for VS Code

The key config file: .vscode/mcp.json at the workspace root. This file is committed to the GitHub repo and shared with all team members. Each user provides their own API credentials via VS Code "inputs" (prompted securely — never hardcoded in the file). The root key is servers, not mcpServers — different from Claude Desktop.

.vscode/mcp.json — Starter config for this system
{
  "servers": {
    "glean": {
      "url": "https://your-org.glean.com/mcp",
      "type": "http",
      "headers": {
        "Authorization": "Bearer ${input:glean_token}"
      }
    },
    "confluence": {
      "command": "npx",
      "args": ["-y", "@confluence/mcp-server"],
      "env": {
        "CONFLUENCE_URL": "${input:confluence_url}",
        "CONFLUENCE_TOKEN": "${input:confluence_token}"
      }
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "${input:github_pat}" }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "."]
    }
  },
  "inputs": [
    { "type": "promptString", "id": "glean_token", 
      "description": "Glean API token", "password": true },
    { "type": "promptString", "id": "confluence_url", 
      "description": "Confluence base URL" },
    { "type": "promptString", "id": "confluence_token", 
      "description": "Confluence API token", "password": true },
    { "type": "promptString", "id": "github_pat", 
      "description": "GitHub Personal Access Token", "password": true }
  ]
}

128-tool threshold: VS Code introduced a 128-tool threshold per request (July 2025, v1.103). When exceeded, VS Code now automatically groups tools into "virtual tool directories" that activate on demand — so the hard ceiling is effectively removed in practice. With Glean, Confluence, GitHub, and filesystem each exposing dozens of tools, you may see this grouping kick in. The "virtual tools" behaviour is on by default; no configuration needed. Document the recommended active-tools set in memory/tools.md for each workflow type.

GitHub Integration Pattern

1

Team repository structure

Create a shared skills repo (e.g. product-community/personal-os-shared) containing the team-level files: shared SKILL.md files, CLAUDE.md baseline, mcp.json, and templates. Each team member clones this as a base and adds their personal files on top. The shared repo is maintained by the Head of Product Ops.

2

Personal workspace on top

Each team member's workspace adds personal context files (context/me.md, context/active.md, personal project files). These are gitignored in the shared repo. A CLAUDE.local.md file handles personal overrides without conflicting with the shared baseline.

3

Skill versioning via git tags

Use git tags on the shared skills repo (v1.0, v1.1) for skill versions. Team members can stay on a known-good version or opt into the latest. This mirrors how software dependencies work — familiarity for the technical setup person; invisibility for end users.

4

Confluence as publishing target

Configure a MCP-triggered workflow: "When I complete a project brief or decision log, use the Confluence MCP to publish it to the [Product Ops > Decisions] space." This creates a bidirectional flow — Confluence as source of truth for shared knowledge, VS Code workspace as AI-enhanced working surface.

File-based (recommended)

Plain markdown files in the workspace. Zero infrastructure, zero cost, fully portable. The AI reads and writes files directly. Works with every VS Code AI assistant. The only persistence model that satisfies the "no custom code" constraint.

Use this
In-context only

Everything in the active session context window. Fast and zero-maintenance, but completely volatile. Suitable for V0 exploration; not a production pattern for a knowledge system.

V0 only

Section 07

Replication & Onboarding

How to take a Personal OS built for one person and roll it out to a non-technical team of ~10. The answer is a three-stage model: prove → package → propagate.

1

Stage 1: Prove (Weeks 1–4, solo)

Build and use the system yourself. Document what works, what gets abandoned, and what you wish existed. This is the "alpha" — you're the only user and the validation test. Track: which files do you actually read? Which prompts do you use daily? Which MCP tools provide real value vs. noise? The output of this stage is a pruned, honest system — not an aspirational architecture.

2

Stage 2: Package (Week 5–6)

Extract the proven elements into a team-shareable repo. Write a single-page README that answers: what is this, why does it exist, how do I get started in 30 minutes? Create a setup checklist covering: clone repo, add context/me.md, configure mcp.json credentials, run one test prompt. Remove anything that requires more than copy-paste to install. The goal is a "30-minute onboarding" experience — no training session needed.

3

Stage 3: Propagate (Week 7+, pilot with 2–3 people)

Onboard 2–3 willing early adopters from the Product Community. Run them through the 30-minute setup guide and observe where they get stuck. Fix the friction before expanding to the full team. Key principle from KM adoption research: pilot users who had a good onboarding experience become internal advocates — the most effective distribution channel for the full rollout.

The 30-minute onboarding flow (non-technical user):

  1. Clone the shared team repo into a local folder
  2. Open the folder in VS Code
  3. Fill in context/me.md using the provided template (5 bullet points: role, priorities, working style, team, key tools)
  4. Enter MCP credentials when VS Code prompts (Glean, Confluence)
  5. Run the test prompt: "Read my context files and tell me what you know about me and my work"
  6. Send first real request: "What should I focus on this week based on my active.md?"

Section 08

Risks & Open Questions

The most likely failure modes for this kind of system — especially for non-technical adopters — and what to validate before committing to a full team rollout.

HIGH Adoption Stall — System Gets Built But Not Used

The #1 failure mode for PKM systems at every level of sophistication. A system built over a weekend that no one uses in month two is a worse outcome than no system. The Personal OS needs daily value within the first week or it will be abandoned. If the first daily interaction doesn't feel faster than the current approach, adoption won't happen.

✦ Validate: Does the morning standup prompt (5 min) feel valuable by Day 3? If not, fix context/active.md before anything else.

HIGH Context Staleness — Memory Files Drift From Reality

As projects evolve and priorities shift, memory files that aren't maintained give the AI wrong context — which is worse than no context. Non-technical users are especially unlikely to proactively update files they don't regularly read. The system needs mechanisms that surface stale content rather than relying on user discipline.

✦ Validate: Add a "staleness check" to the weekly ritual — "Review active.md and learnings.md. Flag any entries older than 30 days that may be outdated." The AI does the work; the user just reviews.

MEDIUM MCP Fragility — Tool Connectivity Breaks the Experience

MCP configuration is brittle. Tool updates, credential expiry, network restrictions, and VS Code version changes all break connections. Non-technical users have low tolerance for configuration debugging. A broken MCP connection mid-workflow destroys trust in the system.

✦ Mitigate: Design the system to be valuable without MCP — local files provide 80% of the value. MCP tools are additive, not load-bearing. Document a "no MCP fallback" mode in the README.

MEDIUM AI Assistant Switching — VS Code AI Landscape is Still Unstable

The brief correctly assumes agnosticism across AI chat assistants. But the SKILL.md ecosystem, CLAUDE.md support, and AGENTS.md support are all relatively new additions to VS Code's Copilot. Feature parity between Copilot, Continue.dev, and other embedded assistants is incomplete. What works in one may not work in another.

✦ Mitigate: Test the system with GitHub Copilot first (best documented support). Keep CLAUDE.md instructions in plain English that any LLM can understand — avoid CLI-specific syntax.

MEDIUM Enterprise Data Governance — What Can Live in Local Markdown?

If team members capture sensitive organisational information (personnel decisions, unreleased product roadmap, commercial terms) in local markdown files that sync to GitHub, there are data governance and security implications. This needs a clear policy before team rollout — not assumed away.

✦ Validate: Establish a "what belongs in the Personal OS vs. Confluence/Glean" policy before onboarding others. Add a "sensitivity check" to CLAUDE.md: "Before writing anything to a file, confirm it doesn't contain information that should stay in [company system]."

LOW Skill File Quality Drift — Team Skills Diverge

Once team members start customising SKILL.md files, the team's shared skill library can diverge. Without a review process, skill quality degrades and sharing value erodes. This is the open-source maintenance problem at small scale.

✦ Mitigate: Monthly skill review as part of the Product Community rituals. Head of Product Ops as skill library owner. Pull request model for skill changes to the shared repo.

LOW Context Window Limits — Large Workspaces Degrade Performance

As the knowledge base grows, loading all context files on every request becomes expensive and potentially exceeds context window limits. The tiered architecture mitigates this significantly — but requires discipline to maintain tier boundaries as the system matures.

✦ Validate: Run a context window usage check after 3 months of use. If CLAUDE.md + auto-loaded files exceed 50k tokens, implement stricter routing and lazy-loading in Tier 1.

The Production Engineering Layer

Three concepts borrowed from the AIDB AgentOS program and Chorouk Malmoum's production AI architecture that upgrade the recommended approach without violating the non-technical constraint. Plus a synthesised 5-layer model adapted from the LinkedIn production stack.

Source context: The AgentOS program (aidbagentos.ai) is a free, self-paced build curriculum from AIDB Training — 12 projects producing portable text files that work across any agentic tool. The production architecture (Chorouk Malmoum, LinkedIn) is a 9-layer RAG application stack whose evaluation, routing, and observability layers translate cleanly to markdown-only patterns. Together they contribute four meaningful upgrades to the core architecture.

Steal 1 — The OS Thought Partner (Helper Agent)

AIDB AgentOS Project 00 introduces a dedicated companion whose only job is to help you build and maintain the OS itself — separate from the AI you use for actual work. This is a structurally important separation that the original architecture missed.

This skill is never invoked for work tasks. It's invoked when you're working on the system — onboarding a new team member, auditing staleness, evolving the architecture. The separation keeps meta-work from contaminating your working context files.

You are the OS architect. Your job is to help maintain, improve, and

## Onboarding Mode
When invoked with "onboard [name]", do the following:

When invoked with "evolve", review the full system and suggest one

Steal 3 — Glean as Lightweight Automation Engine

The AgentOS Automations layer (Project 11) is out of scope for a no-custom-code system. But Glean's MCP capabilities create a passive automation layer without any background processes. The key insight: automation doesn't have to run on a schedule — it can run on demand as part of a deliberate workflow ritual.

1

Consolidation Ritual (Weekly, ~5 minutes)

Invoke the researcher agent: "Consolidate this week's Glean activity into knowledge/this-week.md." Glean's MCP server queries your recent document interactions and people context, the AI synthesises key decisions and open questions, and writes them to a local file. One command replaces 30 minutes of manual note-taking. The local file then serves as a cheap reference point for the entire following week — no Glean queries needed until the next ritual.

2

Project Brain Refresh (Per Project Start)

When opening a project after a gap: "Refresh projects/[name].md using Glean and Confluence." The researcher agent queries both MCP sources for recent changes, diffs them against the existing Project Brain file, and updates only what's changed. Stale context eliminated in 60 seconds. This is the semantic cache pattern from the production stack — "don't re-query what hasn't changed."

3

Stakeholder Map Sync (Monthly)

Invoke the os-helper: "Sync context/org.md with current Glean people data." Glean's people directory via MCP contains org structure, reporting lines, and role changes. The AI updates the stakeholder map in org.md — ensuring that every specialist agent's "who does what" context is accurate without manual maintenance. For a team with regular re-orgs, this alone is a significant time save.

4

Memory Write-Back (Post-Session)

After substantive working sessions, the chief-of-staff agent appends to memory/learnings.md and optionally publishes a structured summary to Confluence via MCP. This closes the loop: context built in VS Code becomes organisational knowledge in Confluence, not just personal memory. Team members searching Confluence later benefit from work done in the Personal OS — the system compounds rather than silos.

The 5-Layer Production Model (Adapted from LinkedIn)

Chorouk Malmoum's 9-layer production AI stack (services/, agents/, prompts/, security/, evaluation/, observability/, .claude/) contains 5 concepts that translate to markdown-only patterns for a non-technical Personal OS. The other 4 layers (infrastructure, containers, tests, frontend) are explicitly out of scope.

Layer 1
Prompts/

Versioned, typed skills — never hardcoded

Production stacks keep prompts as versioned, registered files — not inline strings. In your system: each SKILL.md has a version: field in its frontmatter, and the shared GitHub repo uses git tags for version control. Skills are hot-swappable — update a SKILL.md file and every user on the repo benefits at their next pull. Already in the recommended architecture. The new add: a skills/REGISTRY.md that lists every skill, its version, purpose, and last-updated date — the typed registry that production systems use.

Layer 2
Agents/

Self-correcting routing — document grader + adaptive router

Production agents include a document grader (is this source relevant?) and an adaptive router (route to the right source based on query type). In your system: the memory/tools.md routing table is the lightweight version of adaptive routing. The new add: teach the researcher agent to explicitly score retrieved content — "if Glean returns fewer than 3 relevant results, escalate to a Confluence deep search before synthesising." This is the document grader pattern in plain English, implemented as an instruction in the researcher SKILL.md.

Layer 3
Evaluation/

Golden dataset + quality monitoring — the most skipped layer

Production stacks use a golden test dataset and offline eval pipeline to catch quality regression. Almost no Personal OS implementations include this. The markdown translation: memory/golden-evals.md — 5–10 test questions with known good answers. Run them monthly by pasting the file into a fresh chat session. If response quality has degraded (wrong context, outdated info, wrong tone), the eval pinpoints which memory file is stale. This is the highest-value steal from the production stack — it's the difference between a system that improves and one that quietly drifts.

Layer 4
Observability/

Usage log — per-stage tracing, cost tracking, feedback capture

Production systems trace every query: which stage ran, which source answered, what it cost, was it good? The markdown equivalent: memory/usage-log.md — a lightweight append-only log updated by the chief-of-staff agent after substantive sessions. Format: date, task type, MCP sources used, quality rating (1–5), notes. After 4 weeks of entries, patterns emerge: which MCP sources actually add value, which skills get used, which contexts go stale fastest. This is observability at zero infrastructure cost.

Layer 5
.claude/

Agent context layer — the production stack's Personal OS equivalent

The .claude/rules/ folder in the production stack (code-style.md, testing.md) is exactly the same concept as your Tier 1 constitution — standing instructions that ensure the AI knows the codebase before touching a file. The production stack validates the architecture: every serious AI-assisted workflow ends up here. What the production stack adds: explicit rule files per domain — rather than one CLAUDE.md catching everything, separate rule files for different contexts (writing-rules.md, research-rules.md, communication-rules.md) kept inside a rules/ subfolder. Modular, maintainable, diffable in git.
